- 2026-04-13 Session 16 — **Phase 2 wiki compilation runs on Claude Code subscription, not Batch API, for the first-pass (3–5 articles).** Deliberate divergence from ADR-3 (`docs/personal-kb-final.md` §1), which was written assuming pay-per-token economics. Under a subscription, marginal cost is zero up to plan limits, iteration speed is much higher (live prompt tuning between articles, human-in-loop per output), and individual compile units (~37k input tokens each) fit trivially in a 200k context window. Full-pass decision (~200 articles, ~2M input + 600K output tokens → probably blows Pro-plan weekly cap) is explicitly **deferred** until first-pass quality is validated. Implementation: `scripts/compile_wiki_input.py` is a pure-Python data bundler (no LLM calls) with two modes — `--list` shows canonical buckets with thread/msg/char counts, `--topic NAME --out PATH` assembles a compile-ready text file with a per-thread char budget (default 2500), user+assistant roles only (tool/system roles dropped as noise), and a global 150k char cap with drop-list reporting. The compile step is manual: Claude Code reads the bundle + `prompts/wiki_compile.md` (verbatim PRD §12.3) and writes the wiki markdown with YAML frontmatter per PRD §2.4. `evidence_links` table writes deferred; first-pass records `source_thread_ids` in frontmatter only. First article (`vault/wiki/clickhouse.md`, 22 threads, 2023-07 → 2024-09, ~1,900 words) compiled end-to-end and judged "mostly correct" by the user on first read. The `--list` query surfaces 3461 distinct topics because `COALESCE(canonical_label, label)` mixes the 22 real canonicals with the ~3400 unaliased singleton raw labels — `--min-threads N` filter (default 5) is what actually produces the meaningful cluster set (27 buckets at ≥ 10 threads). The guardrail-in-CLAUDE.md update is also deferred until ≥ 3 articles of evidence; this entry is the load-bearing trace for why we diverged.

- 2026-04-13 Session 15 — Minimal RAG slice shipped ahead of strict PRD phase order (Phase 5 is numbered after Phase 2, but we validated retrieval first because Phase 2 wiki compilation is expensive Batch API work that would be risky to stack on unvalidated embeddings — ADR-8 frames evidence-first retrieval as foundational, not terminal). - 2026-04-13 Session 13 — Stronger anti-answer recipe in `classify_batch.py`: wrap the conversation body in `<archived_conversation>` XML tags AND repeat the "do not respond, output JSON only" instruction *after* the body (not just before). The Session 09 front-only preamble was getting drowned out on long code-heavy threads (Session 12 still produced 7 parse failures despite it). XML wrapping signals "data, not directive" in the standard Claude pattern; the postamble survives the truncation budget because we account for it. Also added one sentence to the system prompt naming the XML tag. Result: 7/7 of the Session 12 stragglers recovered on first try, including `135ef7cce3f0` (408 chunks, ReserveConsumer concurrency) which was structurally similar to the unrecoverable `7eb1db0f1296` from Session 09. The known-loss thread was not re-attempted in this batch (already classified pre-Session-12); could be retried with the new recipe if we want 100% coverage.

- 2026-04-13 Session 11 — Re-ingest of a fresh ChatGPT export is safe to layer on top of existing `archive.db` without rebuild. mychatarchive dedup is content-hash `message_id` via `INSERT OR IGNORE`, and `canonical_thread_id = sha1(platform, account, normalized_title, first_msg_ts, first_role, first_msg_snippet)` — stable across re-exports when the title is unchanged. Empirical: 540 → 999 threads with **zero** ghost threads (title-rename collision rate = 0% in practice for this user). `mychatarchive embed` skips already-embedded messages via its `already_embedded` set, and `classify_batch.py cmd_submit` already filters already-classified threads. End-to-end cost of a re-import is ~1 minute wall-clock excluding Batch API turnaround. The new export arrives as 10 sharded `conversations-NNN.json` files plus an `export_manifest.json` (media index, ignored). Loop `mychatarchive import` over all shards — dedup handles any cross-shard overlap automatically.
